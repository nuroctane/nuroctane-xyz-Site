# Digital Sea — Interactive Scroll Site
### Build brief for Bolt.new

Attach `digital_sea.jpg` to this prompt before sending. Tell Bolt explicitly:
"Use this image for the color palette only: deep teal/petrol background (#0b2730), mid cyan blocks (#1c5f70), bright glow accents (#bdeff2 and #eafcff). Do not infer layout or motion from the image, that is specified below."

---

## 1. Tech stack (pin this, don't let Bolt choose)

- Vite + React
- `three`, `@react-three/fiber`, `@react-three/drei` for the 3D scene
- `gsap` with ScrollTrigger to bind scroll position to camera progress
- `@react-three/postprocessing` for bloom + fog, to get the glow/haze in the reference image
- Bolt runs in an in-browser WebContainer. Stick to these well-known npm packages so install/build doesn't choke.

## 2. Visual language (from the reference image)

- Palette: deep teal/petrol base, mid cyan geometry, bright cyan-white glow on edges and light shafts
- Geometry: translucent rectangular blocks of varying width/height, overlapping like stacked server towers, slight fresnel/glass material so edges catch light
- Volumetric light shafts entering from above/background, soft and diffuse, not hard-edged
- Particles: thin streaks/dots drifting slowly downward, additive blending so they glow rather than look solid
- HUD node style: thin white/cyan circles connected by straight lines, small monospace text labels inside circles or boxes, this is the direct template for the floating UI nodes below
- Faint scanline/noise overlay for a "submerged" feel

## 3. Scene + camera

- Define a Catmull-Rom spline running through the scene representing the path forward through the digital sea
- Camera position and look-direction are driven by scroll progress (0 to 1), not by free orbit controls
- Add a small amount of independent vertical/lateral drift on top of the path so it doesn't feel mechanical, like current pushing the camera slightly off-line
- Block geometry: use `instancedMesh` for performance, scatter blocks along the path with randomized scale/rotation/offset to match the corridor look in the reference image
- Particle system: a points-based field of a few thousand particles drifting downward and slightly toward camera, recycled when they pass camera Z position (standard infinite-particle-loop pattern)

## 4. Scroll mechanism

- Full-viewport fixed `<Canvas>` behind the content
- A tall invisible scroll-spacer div defines total scroll length. With 15 nodes, something like `1500vh` to `1800vh` gives each node enough room to approach, face the user, and recede without feeling rushed, tune by feel once it's running
- Read scroll progress via GSAP ScrollTrigger or a scroll-progress hook, normalize to 0–1
- Map that 0–1 value to the spline's `t` parameter for camera position, and to `t + small lookahead` for the look-target, so the camera always looks slightly ahead along the path

## 5. Floating UI nodes (the core interactive piece)

Each node is anchored to a fixed point in 3D space along the path. Render it as an HTML overlay anchored to that 3D position (drei's `<Html>` component, with `transform` and `occlude` enabled so it scales/hides correctly with depth) styled to match the circle-and-line HUD look from the reference image.

**Card contents, per node:**

- A small circular avatar image (the platform profile picture)
- A larger rectangular preview image (a cropped capture of the profile page, header/bio area only, not the full scrollable page, so it stays legible at card scale)
- Platform label + handle
- One short stat line (bio, follower count, or similar)
- All of this wrapped in the thin cyan circuit-line border treatment from the reference image

Both images are static assets shipped with the project (see section 6), not fetched live at runtime. No CORS proxy, no API keys in the client, no broken-image risk if a platform changes its markup later.

**State logic, per node, per frame:**

1. Compute `proximity` = how close the current scroll progress is to that node's defined scroll range (`scrollStart` to `scrollEnd`), normalized 0 to 1.
2. At `proximity = 0`: node sits in an idle resting rotation (e.g. angled away from camera, as if drifting in the sea), low opacity, smaller scale.
3. As `proximity` rises toward 1: slerp the node's rotation from its idle rotation toward `lookAt(camera.position)`, and ramp opacity and scale up toward full/legible.
4. At `proximity ≈ 1`: node is fully facing the camera, fully opaque, fully sized, and clickable. Clicking opens `node.url` in a new tab.
5. As scroll moves past the node, reverse the interpolation so it fades and turns away again, don't just snap.

This is a single `useFrame` loop iterating over the node list, each node carrying its own scroll range and idle rotation, no per-node custom code needed.

## 6. Image sourcing, what's actually automatable

Tested live against the nine accounts. Five platforms expose a clean profile picture via standard og:image metadata, no login needed. Four block automated access outright (robots.txt or bot detection), the same wall a backend scraper or screenshot service would hit in production.

**Avatar fetchable, no auth:** YouTube, SoundCloud, Substack, Twitch, Kick
**Blocked at the page level:** Instagram, X, TikTok, AniList

None of the five "working" platforms expose a full page screenshot, only the avatar and some metadata. A real preview image, avatar or page screenshot, is a manual capture for all nine, for two reasons: consistency (nine wildly different native page styles won't read well as a uniform card design) and reliability (no scraping pipeline to maintain, no broken images when a platform redesigns).

**For each of the nine, export two images and drop them into `/public/assets/nodes/`:**
1. `{platform}-avatar.png`, square, the profile picture
2. `{platform}-preview.png`, a cropped screenshot of just the header/bio area of the profile page

These are captures of your own accounts, so there's no rights issue. A quick screenshot in a normal logged-in browser tab works fine, even for Instagram, X, and TikTok where automated fetching is blocked.

AniList does have a public GraphQL API (`https://graphql.anilist.co`) that returns avatar, banner, and stats with no auth required, if you ever want that one node to refresh itself automatically later. Not needed for the first build.

## 8. Performance note for Bolt

Flag explicitly: on lower-end devices, drop particle count and disable postprocessing bloom, falling back to a simpler gradient + static blocks. A full bloom/postprocessing R3F scene can choke on mobile GPUs if left unconstrained.

## 9. Optional: real video as fallback/poster

If a Higgsfield loop gets generated, it can sit as a `<video>` poster/background layer that's visible before the 3D scene finishes loading, or as a fallback for devices that can't run WebGL well. It is not a reference for the 3D logic above, that logic is fully specified in this document.
